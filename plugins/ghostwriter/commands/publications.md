# Publications Management

Manage publications in `.ghostwriter/publications/`.

A publication represents a writing project — docs site, blog, book, newsletter, etc. Each publication has a config pointing to where content lives, an associated author, and accumulated learned patterns.

## Usage

```bash
/publications                    # List all publications
/publications list               # Same as above
/publications add                # Add a new publication (interactive)
/publications remove <slug>      # Remove a publication
/publications show <slug>        # Show publication details
```

## Prerequisites

Ensure `.ghostwriter/` exists. If not, run `/setup` first.

## Commands

### list (default)

List all publications in `.ghostwriter/publications/`.

For each publication, read `config.yml` and extract: `publication.name`, `publication.media`, `content.root`, and the author name from the preset chain.

Display:

```
Publications:

  developer-docs
    Name: Developer Documentation
    Media: docs
    Author: Acme Engineering
    Content: apps/docs/src/content/docs/**/*.md
    Pipeline runs: 3

  company-blog
    Name: Company Blog
    Media: blog
    Author: Jane Doe
    Content: content/blog/**/*.md
    Pipeline runs: 0
```

If no publications exist, display: "No publications found. Run `/publications add` or `/setup` to create one."

### add

Interactive publication creation. This reuses the setup wizard's publication flow.

**Pre-check:** Ensure at least one author exists in `.ghostwriter/authors/`. If none, prompt: "No authors found. Let's create one first." Then run the author creation flow (same as `/authors add`).

Ask these questions ONE AT A TIME:

1. "What's the name of this publication?" (e.g., "Developer Docs", "My Tech Blog")

2. "Which author should write this?" → list existing authors, plus "Create new author"

3. "What type of writing is this?"
   - Blog, Book, Press, Email, Academic, Docs

4. Choose style based on medium (same options as setup wizard)

5. "Where does your content live? (path to directory with .md files)"
   - e.g., `apps/docs/src/content/docs`, `manuscript/`, `content/blog/`

6. "What file types should be processed? (default: **/*.md)"
   - User can provide custom glob like `**/*.{md,mdx}`

7. "Should files be edited in-place, or do you want a staged workflow?"
   - In-place (default for docs, email, press)
   - Staged: ask for stage names or use defaults (original, revised, humanized, final)

Slugify the publication name.

Create the publication:

```bash
mkdir -p ".ghostwriter/publications/{pub-slug}/pipeline"
```

Determine the preset name from medium/style (same mapping as setup wizard):

| Medium | Style | Preset |
|--------|-------|--------|
| Blog | personal | casual |
| Book | fiction | book |
| Book | nonfiction | book |
| Book | technical | technical-book |
| Book | children | casual |
| Press | news | business |
| Press | opinion | book |
| Email | newsletter | casual |
| Email | business | business |
| Academic | paper | academic |
| Docs | technical | technical-docs |

Compute `content.root` as relative path from `.ghostwriter/publications/{pub-slug}/` to the user-provided content path.

Write `.ghostwriter/publications/{pub-slug}/config.yml`:

```yaml
# {Publication Name} — Writing Config
# Generated by /publications add

version: '1.0'

presets:
  - ../../authors/{author-slug}/author-persona.yml
  - {preset-name}

publication:
  name: "{Publication Name}"
  media: {media}
  audience: {from template}

content:
  root: {relative-path-to-content}
  glob: "{user-provided glob or **/*.md}"
  # stages:           # Uncomment for staged workflow
  #   - original
  #   - humanized
  #   - final

writing_style:
  tone: {from template}
  formality: {from template}
  voice: {from template}

# Detection Rules — uncomment to override preset defaults
# rules:
#   punctuation:
#     em_dash:
#       ai_threshold: 4.0
```

If staged workflow was chosen, uncomment and populate the stages field.

Write `.ghostwriter/publications/{pub-slug}/learned-patterns.md`:
```markdown
# Learned Patterns: {Publication Name}

Patterns specific to this publication.
Updated automatically by the ai-engineer during humanization rounds.
```

Display:
```
Publication '{name}' created!

Config:  .ghostwriter/publications/{pub-slug}/config.yml
Content: {content-path}/**/*.md
Author:  {author-name}

Quick start:
  /analyze <file> {pub-slug}
  /humanize <file> {pub-slug}
  /humanize-all {pub-slug}
```

### remove

```bash
/publications remove <slug>
```

Confirm: "Remove publication '{slug}'? This deletes the config, learned patterns, and all pipeline data. Content at {content.root} is NOT affected. (y/n)"

If confirmed:
```bash
rm -r ".ghostwriter/publications/{slug}"
```

Display: "Publication '{slug}' removed. Content at {content.root} was not modified."

### show

```bash
/publications show <slug>
```

Read `.ghostwriter/publications/{slug}/config.yml` and display:

```
Publication: {name} ({slug})

Config:  .ghostwriter/publications/{slug}/config.yml
Media:   {media}
Author:  {author name} ({author slug})
Preset:  {preset name}

Content:
  Root:    {resolved absolute path}
  Glob:    {glob}
  Stages:  {stages or "in-place"}
  Files:   {count of matching files}

Pipeline:
  Runs:    {total run count across all files}
  Latest:  {latest run path and date}

Learned patterns: {N} patterns
  File: .ghostwriter/publications/{slug}/learned-patterns.md
```

If the slug doesn't exist, display: "Publication '{slug}' not found. Run `/publications list` to see available publications."
